§ THE BASICS

MCP is the OpenAPI for your agents

Let's start from the basics. MCP - Model Context Protocol - is the OpenAPI for your agents: one standard contract between any AI application and any external system. It was born from a real need inside Anthropic - every integration was bespoke, the classic N-times-M problem - and open-sourced in November 2024. 2025 was the adoption year: ChatGPT, Cursor, Gemini, Copilot, VS Code - everyone speaks it. And in December, Anthropic donated MCP to the Agentic AI Foundation, a directed fund under the Linux Foundation co-founded with Block and OpenAI. It's not a vendor protocol anymore; it's infrastructure. Today: the Python SDK alone is at about 300 million downloads a month. And in two weeks, the largest revision since launch ships - we'll get there.
§ WHY IT WON

One agent, any server

Why did it win? This picture. Your agent in the middle, and everything it touches around it - GitHub, Postgres, Slack, your filesystem, Logfire, your internal API. Every wire is the same protocol. Before MCP this was the N-times-M problem: every agent times every tool meant a custom integration - function calling glue per provider, a plugin format per framework. MCP collapses it: a server is written once and works with any client; a client speaks MCP and gets the whole ecosystem for free. Locally a server runs as a subprocess over stdio; remote ones are plain HTTP endpoints with OAuth. Same messages either way - but the point is this shape: one agent, any server.
§ THE QUESTION IN YOUR HEAD

Why not just a CLI?

The question half this room is thinking: my agent runs bash - why not just give it a CLI? Honest answer: often you should! Models are excellent at text, and a local trusted one-shot task needs no protocol. But four things a CLI can't give you. A typed, discoverable contract - the model gets JSON schemas, not --help scraping. Interaction mid-call - a tool can ask the user a question and resume; a CLI can only fail. Auth - an MCP server does OAuth on the user's behalf; a CLI wants your credentials in env vars. And remote - a server can run far away from the agent, multi-tenant, observable, with structured results and streaming. So: not either-or. Text-shaped local work: CLI. Types, auth, interaction, remote: that's what MCP is for.

§ THE GROUND IS MOVING · SPEC 2026-07-28

July 28: the largest revision since launch - and it's a subtraction

Server Acan answer ✓
Server Bcan answer ✓

Stateless by default: no sessions, no initialize - server/discover replaces the handshake. Any node answers any request.

On July 28 the largest spec revision since launch ships - RC locked in May, and the headline is what got deleted, not what got added. The protocol becomes stateless by default. Every request is self-contained: version and capabilities travel in _meta. The initialize handshake is replaced by server/discover. What that buys you is this diagram: any node can answer any request. No sticky sessions, no session stores, no resumability machinery. You deploy an MCP server like any HTTP service. Best practice starting today: write handlers that assume nothing survives between requests. If your logic lives in plain functions behind the MCP layer, this costs you nothing.

The redline. Not deprecated - removed. Sessions gone, handshake gone, ping gone, setLevel gone, SSE resumability gone, tasks moved out to an extension. The good news: most of this is SDK plumbing - v2 handles both eras for you and you'll barely notice. The part that IS your problem: sampling, roots and logging are deprecated with a formal lifecycle - at least twelve months, then removal. If your product depends on them, plan now: call your LLM provider directly, pass paths as arguments, log to stderr or OpenTelemetry. Elicitation stays - and got better. Next slide.

Before I show you the new way, here's how an interactive tool works today, on the current spec. The client calls the tool and the HTTP stream stays open. Mid-call, the server turns around and initiates its own request - elicitation/create, "which units?" - pushed back over that open stream. The answer has to come back to the same session, on the same node that's holding the call. On stdio this is fine - one process, one pipe. Over HTTP at scale it's the sticky-session problem from two slides ago: sessions, resumable streams, one exact node that knows you. That server-initiated callback is exactly what the new revision removes. So how can a tool still ask questions? Next slide.
INTERACTIVE TOOLS · FROM JULY 28 · SEP-2322

MRTR: the server can still ask

Multi round-trip requests, my favorite part of the new spec. A tool that needs input doesn't push over a side channel - it returns early with input_required and the question embedded. The client answers and re-calls with an opaque requestState. All state travels in the request, so round two can hit a different node. And here's the SDK v2 payoff: resolver dependency injection. You declare what you need as a typed parameter; the SDK asks the user - over classic elicitation on 2025 sessions, over MRTR on 2026 sessions. Same tool code, both protocol eras. Statelessness didn't cost interactivity. It moved it into the type system, where it belongs.
§ THE V2 API · RESOLVERS

Declare what you need - the SDK asks

async def confirm_backorder(title: str) -> Backorder | Elicit[Backorder]:
    if stock(title) > 0:
        return Backorder(confirm=True)
    return Elicit(f"{title!r} is out of stock. Order anyway?", Backorder)

@mcp.tool()
async def order_book(
    title: str,
    backorder: Annotated[Backorder, Resolve(confirm_backorder)],
) -> str:
    if not backorder.confirm:
        return "No order placed."
    return f"Ordered {title!r}."
And this is what that looks like in v2 - straight from the docs, lightly trimmed. The tool declares backorder as a typed parameter with a Resolve annotation. The resolver runs server-side; if it can compute the answer, it just returns it. If it needs the user, it returns Elicit with a message and a schema - and the SDK handles the round trip: classic elicitation on a 2025 session, input_required plus requestState on a 2026 session. Your tool body never knows the difference - by the time it runs, backorder is just a validated Pydantic model. One caveat: this is beta API - b2 today - so details may still shift slightly before stable.

§ PROVE IT WORKS · THE HILL I DIE ON

Test your server - in memory

from mcp import Client
from server import mcp

@pytest.fixture
async def client():
    async with Client(mcp, raise_exceptions=True) as c:
        yield c

async def test_add(client: Client):
    result = await client.call_tool("add", {"a": 1, "b": 2})
    assert result.structured_content == {"result": 3}
The hill I will die on. Reviewing community servers, almost nobody tests them - people spin up an inspector, click a tool once, ship. In v2 this is the whole pattern: Client takes your server object directly, in memory. No transport, no subprocess, no fixtures with ports. raise_exceptions=True gives you real tracebacks instead of sanitized protocol errors. Assert on structured_content for the data, snapshot-test the full result if you want belt and suspenders. It runs in CI like any other test. And notice: if you kept logic at the edge, you barely need this - you unit-test the functions and keep a handful of these as contract tests.
§ THE LOW-LEVEL SERVER · V1

v1: decorators and magic

from mcp.server.lowlevel import Server
from mcp.types import TextContent, Tool

server = Server("echo")

@server.list_tools()
async def list_tools() -> list[Tool]:
    return [Tool(name="echo", inputSchema=SCHEMA)]

@server.call_tool()
async def call_tool(name: str, args: dict) -> list[TextContent]:
    return [TextContent(type="text", text=args["text"])]
For those on the low-level Server, here's a full minimal v1 server: an echo tool. Registration happens through decorators on the server instance. Fields are camelCase - inputSchema - and handlers return bare lists that the SDK wraps into result types for you, magically. Convenient, but implicit: the wrapping rules were never obvious, and the decorator coupling made the internals hard to evolve. Now the same server in v2.
§ THE LOW-LEVEL SERVER · V2

v2: constructor parameters, no magic

from mcp.server import Server, ServerRequestContext
import mcp_types as t

async def list_tools(
    ctx: ServerRequestContext,
    params: t.PaginatedRequestParams | None,
) -> t.ListToolsResult:
    return t.ListToolsResult(tools=[t.Tool(name="echo", input_schema=SCHEMA)])

async def call_tool(
    ctx: ServerRequestContext,
    params: t.CallToolRequestParams,
) -> t.CallToolResult:
    text = params.arguments["text"]
    return t.CallToolResult(content=[t.TextContent(type="text", text=text)])

server = Server(
    "echo",
    on_list_tools=list_tools,
    on_call_tool=call_tool,
)
The same echo server in v2. Handlers are plain functions, registered as constructor parameters - on_list_tools, on_call_tool. Everything is typed: a ServerRequestContext plus typed params in, a full result type out - ListToolsResult, CallToolResult. No auto-wrapping: what you return is what goes on the wire, validated against the negotiated spec version. Fields are snake_case, the types come from the standalone mcp_types package. More explicit, slightly more verbose - and it's what made the stateless dispatcher core possible.

The change I personally fought for. SEP-414: W3C trace context in _meta, so you get one distributed trace across your agent, the client, and every server it calls. In v2 the docs' own words are "your server is already traced" - every message gets an OpenTelemetry span, and without an exporter it's a no-op, so it costs nothing. Install logfire, call configure, and the traces just appear. Any OTel backend works - the Pydantic bias is showing again. The number to remember: ten months from proposal to acceptance. That's not dysfunction - that's what keeping a standard boring costs.
§ OBSERVABILITY · IN PRACTICE

Instrumenting with Logfire: one line less

v1 · today

from mcp.server.fastmcp import FastMCP
import logfire

logfire.configure()
logfire.instrument_mcp()

mcp = FastMCP("Bookshop")

@mcp.tool()
def search_books(query: str) -> str: ...

v2 · from July 28

from mcp.server import MCPServer
import logfire

logfire.configure()


mcp = MCPServer("Bookshop")

@mcp.tool()
def search_books(query: str) -> str: ...
In practice, with Logfire. On v1 you configure Logfire and call logfire.instrument_mcp() - that patches the SDK to emit spans. On v2, spot the difference: instrument_mcp is gone. The SDK emits OpenTelemetry spans natively - every message, out of the box, a no-op if you have no exporter. logfire.configure() alone picks them up, and any OTel backend works the same way. Instrumentation you delete instead of add - my favorite kind.

Where the SDK is today: second beta shipped July 14, stable v2 targeted for July 27 - the day before the spec. uv add mcp still resolves to 1.x; betas are strictly opt-in, and pin exact versions. The renames: FastMCP is now MCPServer - same decorator API, look at the code, your signature is the schema. Fields are snake_case. The wire types live in a standalone mcp-types package that only depends on pydantic - build tooling without dragging the transport stack. And the HTTP stack moved from httpx to httpx2 - which I maintain, so consider the bias disclosed. SSE support built in, OS trust store for TLS.
